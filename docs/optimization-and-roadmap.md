# 优化与长期规划

本文档整理针对 **math-demo**（Word 试卷解析 → Parser / API / Web）的后续优化方向与中长期规划，便于排期与讨论。权威架构说明仍以根目录 [README.md](../README.md) 为准。

---

## 一、近期可做的优化（性价比高）

### 1. 解析与导出稳定性

- 大文件超时、失败重试与明确错误码；前端提示可统一为错误分类（例如：可少选题目、稍后重试、超时加大 `EXPORT_TIMEOUT_MS` 等）。
- Parser 导出依赖 `data/images` 下多 `file_id` 目录时，可评估旧文件清理策略或磁盘占用说明，避免长期堆积。

### 2. 状态与数据

- 当前多卷解析与题目缓存依赖 API 进程内内存（如 `parseProgressStore`），重启后丢失。若产品需要「隔天仍可导出」，可引入短期持久化（SQLite / Redis / 文件 JSON）；若定位为「会话内有效」，应在 UI 明确提示。

### 3. 前端体验

- 多文件同时解析时，可在每卷卡片内保留 skeleton，全局仅用轻量提示，降低「页面卡住」的误判。
- 组卷预览拖拽可后续考虑专用排序组件或无障碍（键盘操作），减少误拖。

**已落实（上传页 / 组卷）：** 顶部 `el-alert` 显示并行任务数与说明；题目卷内 skeleton 附简短提示；HTML 模式解析中展示骨架占位；单卷解析完成后平滑滚动至对应卡片；组卷区工具条（已选题数 +「清空已选」确认）；轮询最长约 15 分钟自动失败提示；预览拖拽时目标行高亮与 `Rank` 手柄图标；上传区在解析中轻微高亮且 HTML 模式下禁止重复提交直至完成。

### 4. 类型与契约

- `packages/shared-types` 与 `apps/web`、`apps/api` 的 DTO 建议以共享包为单一事实来源，避免三处手写漂移。

### 5. 测试

- **Parser**：覆盖 `/export` 的 `use_questions_payload_order`、跨卷同 `question.id`（如 `q_0`）场景。
- **API**：`assembly` 组卷、缓存缺失时的错误响应。
- **Web**：E2E 主路径「上传 → 跨卷选题 → 预览调序 → 导出 Word」。

### 6. 可观测性

- 统一 request id，使 API 与 Parser 日志可关联；记录导出耗时、题目数、是否命中缓存等，便于后续扩容与排障。

## 二、中长期规划（方向性）

### 1. 产品能力

- 题库维度：标签、难度、知识点；组卷模板（题型结构、分值）。
- 若面向学校/机构：多用户、试卷归属与权限。
- 输入形态扩展：除 Word 外，PDF 入题、LaTeX 直编等（与现有 OMML/公式图链路并存需单独设计）。

### 2. 解析质量

- 低置信度题目的交互式校正；人工标注回流（规则库或模型迭代）。
- WMF/公式图与 DPI、字体一致性持续优化；可将 `tools/wmf-gdi-render` 等纳入 CI 或发布前检查。

### 3. 架构

- 解析与导出异步化：队列 + Worker，上传返回 `job_id`，前端轮询或 WebSocket，避免长 HTTP 占用连接。
- 对象存储（S3 兼容）存放原始 docx 与导出结果，便于多实例与横向扩展。

### 4. 工程化

- 修复 Monorepo 内 API `tsc` 等与 Express 类型相关的构建问题，使 CI 严格通过。
- 完善 Docker Compose 一键环境（含 Parser + API + Web），降低新成员上手成本。

---

## 三、单人维护时的建议优先级

1. **会话/缓存策略与 UI 提示**（避免用户误以为「丢数据」或「必须一次导完」）。
2. **主路径 E2E + Parser 导出关键单元测试**（防回归）。
3. **异步任务与超时/进度体验**（文件变大后优先受益）。

有稳定用户画像后，再投入题库、权限与对象存储等较重能力。
